<objective>
Convert remaining fmt::format() calls in the draw pipeline to fmt::format_to() to eliminate intermediate string allocations, and replace under-estimated out.reserve() calls with escape-aware size estimates that prevent repeated reallocations during frame rendering.

Purpose: Each fmt::format() call allocates and returns an intermediate std::string that is then appended to the output buffer. Converting to fmt::format_to() appends directly, eliminating 13 intermediate allocations per frame. The reserve estimates (currently width*height) are 10-20x too low because they don't account for ANSI escape codes (~12-20 bytes per visible character), causing O(n) reallocations as the string grows.

Output: Modified btop_draw.cpp with zero intermediate format allocations and right-sized output buffer reserves.
</objective>

**Pattern A: `out += fmt::format(...)` — direct conversion to format_to**

For calls where the result is directly appended to `out`, convert to `fmt::format_to(std::back_inserter(out), ...)`.

Apply to these locations:
- **Line 291** (createBox): `out += fmt::format("{}{}{}{}{}{}{}{}{}", ...)` → `fmt::format_to(std::back_inserter(out), "{}{}{}{}{}{}{}{}{}", ...);`
- **Line 297** (createBox): `out += fmt::format("{}{}{}{}{}", ...)` → `fmt::format_to(std::back_inserter(out), "{}{}{}{}{}", ...);`

**Pattern B: Result assigned to a local variable — convert to format_to with local string**

For calls where the result is stored in a variable and then used in concatenation, use `fmt::format_to` with a local string and `std::back_inserter`. This still avoids the fmt::format internal allocation overhead while keeping the variable available.

Apply to these locations:

<task type="auto">
  <name>Task 2: Update draw function reserve() estimates with escape-aware multipliers</name>
  <files>src/btop_draw.cpp</files>
  <action>
Update the 5 `out.reserve()` calls in draw functions to account for ANSI escape code overhead. Each visible character position typically requires 12-20 bytes of escape codes (cursor movement, color changes, style resets).

**Line 588 (Cpu::draw):**
Change: `out.reserve(width * height);`
To: `out.reserve(width * height * 16);`
Rationale: Cpu::draw has moderate escape density — color per core bar, temperature colors, load average formatting. Multiplier 16 accounts for typical `\033[38;2;R;G;Bm` (19 bytes) per color change with ~1 color per 2-3 visible characters, plus cursor movement.

**Line 1042 (Mem::draw):**
Change: `out.reserve(width * height);`
To: `out.reserve(width * height * 12);`
Rationale: Mem::draw is moderately sparse — graph rendering plus a few status lines. Lower multiplier than Cpu/Proc.

**Line 1218 (Net::draw):**
Change: `out.reserve(height * width);`
To: `out.reserve(height * width * 10);`
Rationale: Net::draw is the sparsest draw function — primarily graphs with relatively few color changes per line. Lowest multiplier.

**Line 1479 (Proc::draw):**
Change: `out.reserve(width * height);`
To: `out.reserve(width * height * 18);`
Rationale: Proc::draw has the densest output — per-process rows with multiple color changes (PID, name, CPU%, MEM%, state), cursor positioning for each column. Highest multiplier.

**Line 1702 (Proc::draw detailed section):**
Change: `out.reserve(width * height);`
To: `out.reserve(width * height * 16);`
Rationale: Detailed process view has similar density to Cpu with additional info sections.

**Add comments documenting the multiplier basis:**
For each reserve call, add a brief inline comment:
```cpp
out.reserve(width * height * 16);  // ~16 bytes/visible char: escape codes + cursor moves
```

**IMPORTANT:** These are conservative estimates (intentionally over-reserving slightly). The trade-off is a single larger allocation vs. many reallocations. For a typical 200x50 terminal:
- Old: 10,000 bytes reserved, actual output ~100-300KB → 15-30 reallocations
- New: 160,000-180,000 bytes reserved → 0-2 reallocations
This is ~160-180KB per draw function, which is acceptable for a TUI that redraws every 2 seconds.
