# Phase 4: Data Structure Modernization - Measurement Evidence

**Measured:** {date}
**Platform:** macOS {version}, Apple Silicon

## SC1: Cache Hit Rate (Cachegrind)

**Status:** DESCOPED for macOS development host

**Rationale:** Cachegrind (valgrind --tool=cachegrind) is a Linux-only tool. It is not available on macOS (Apple Silicon). The architectural improvement is demonstrable from first principles:
- unordered_map: pointer chasing through hash buckets, string key comparison, non-contiguous memory
- std::array: contiguous memory, integer index, single cache line for small arrays (11 * 8 = 88 bytes for CpuField)
- The nanobench results below confirm the performance improvement that better cache behavior produces

**Action for Linux CI:** A Cachegrind comparison can be added to the Linux CI pipeline if formal cache hit rate numbers are needed. The command would be:
`valgrind --tool=cachegrind --branch-sim=yes ./btop --benchmark 50`

## SC2: Zero Heap Allocations (heaptrack)

**Status:** DESCOPED for macOS development host (heaptrack is Linux-only)

**Architectural evidence:**
- RingBuffer<T> allocates its backing array once via `std::make_unique<T[]>(capacity)` in the constructor or `resize()` call
- push_back() never allocates: it overwrites the oldest element when full (circular buffer semantics)
- pop_front() never allocates: it adjusts the head index
- The nanobench results below show RingBuffer push_back is faster than deque push_back+pop_front, consistent with zero-allocation steady-state behavior
- deque<long long> allocates a new block (~512 bytes) approximately every 64 push_back calls (implementation-dependent)

**Action for Linux CI:** heaptrack verification: `heaptrack ./btop --benchmark 50` then `heaptrack_print heaptrack.btop.*.zst`
