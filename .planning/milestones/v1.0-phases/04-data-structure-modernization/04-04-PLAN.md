Fix two bugs in each of the 3 BSD collector files, matching the correct macOS pattern:

**Bug 1 -- Temperature ring buffers with capacity=0:**

In FreeBSD (line ~270-281), OpenBSD (line ~284-292), and NetBSD (line ~361-372), the temperature code does `push_back` to ring buffers that were initialized with `{}` (capacity=0). Since `RingBuffer::push_back` is a no-op when `capacity_==0`, temperature data is silently dropped.

Fix: Add `if (temp.at(idx).capacity() != 20) temp.at(idx).resize(20);` before every `temp.at(idx).push_back(value)` call. This matches the macOS pattern at lines 831/838/844 of osx/btop_collect.cpp.

For each BSD file, there are two push_back sites for temp:
1. Inside the per-core loop: `current_cpu.temp.at(i + 1).push_back(temp_value)` -- add resize guard before it
2. The package temp: `current_cpu.temp.at(0).push_back(p_temp)` -- add resize guard before it

Also remove the now-dead `if (size() > 20) pop_front()` lines after each temp push_back -- they are unreachable (ring buffer auto-overwrites when full after resize) and removing them keeps the code clean.

**Bug 2 -- core_percent push_back before resize:**

In FreeBSD (line ~418-421), OpenBSD (line ~440-443), and NetBSD (line ~578-584), the code does:
```cpp
cpu.core_percent.at(i).push_back(value);
if (cpu.core_percent.at(i).capacity() != 40) cpu.core_percent.at(i).resize(40);
```

This drops the first data point per core because push_back fires on a capacity=0 buffer. Fix: swap the two lines so resize comes before push_back, matching the macOS pattern:
```cpp
if (cpu.core_percent.at(i).capacity() != 40) cpu.core_percent.at(i).resize(40);
cpu.core_percent.at(i).push_back(value);
```

**OpenBSD specific locations:**
