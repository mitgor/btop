  <name>Task 1: Create macOS CPU sampling profiler script</name>
  <files>scripts/profile_hotfunctions.sh</files>
  <action>
Create a bash script `scripts/profile_hotfunctions.sh` that automates CPU sampling profiling on macOS. The script must:

1. **Check prerequisites:** Verify `build-bench/btop` binary exists (built with `-DBTOP_BENCHMARKS=ON`). If not, print instructions and exit 1.

2. **Launch btop in benchmark mode:** Run `./build-bench/btop --benchmark 20` in the background (this runs 20 collect+draw cycles producing realistic CPU workload without needing a terminal UI).

3. **Attach macOS `sample` profiler:** Use `sample <PID> 5 -f /tmp/btop_profile.txt` to capture 5 seconds of CPU sampling data from the btop process. The `sample` command is macOS-native and requires no special privileges for processes owned by the current user.

4. **Parse the sample output:** After sampling completes, parse `/tmp/btop_profile.txt` to extract the call graph with hit counts. The `sample` tool outputs a call tree with hit counts per function. Extract top-level functions, sum their hit counts, compute percentage of total samples, and output a ranked table sorted by CPU %.

5. **Output format:** Print a markdown table to stdout:
```
| Rank | Function | Samples | CPU % |
|------|----------|---------|-------|
| 1    | func_name | 1234   | 45.2% |
```

6. **Also save raw output:** Keep `/tmp/btop_profile.txt` for manual inspection.

The script should be executable (`chmod +x`). Use `#!/usr/bin/env bash` and `set -euo pipefail`.

Important: The `sample` command on macOS outputs a tree-format call graph. To get a flat function ranking, parse the "Sort by top of stack" section near the end of the output (sample provides this automatically). Lines look like:
```
    sort_function_name    1234
```
Parse these lines, compute total, and derive percentages.

If `--benchmark 20` finishes before sampling completes (it runs fast), increase to `--benchmark 100` or use a fallback: launch btop normally in a subshell with a timeout, then sample it. Prefer the benchmark mode approach since it does not require a terminal.

Handle the race condition: start btop first, wait 0.5s for it to begin work, then start sampling. Use `wait` or check that btop is still running before sampling.
