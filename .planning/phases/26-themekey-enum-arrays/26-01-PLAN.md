## Objective

Replace `Theme::colors` (unordered_map<string, string>), `Theme::rgbs` (unordered_map<string, array<int,3>>), and `Theme::gradients` (unordered_map<string, array<string,101>>) with enum-indexed `std::array` containers. Define a `ColorKey` enum for the ~40 fixed color names and a `GradientKey` enum for the ~14 gradient base names. Update the accessor functions `c()`, `g()`, `dec()` to take enum parameters. Provide backward-compatible string-accepting overloads for the Meter/Graph classes (which store gradient names as strings -- migrated in Plan 02).

## Context

Read these files before starting:

- `src/btop_theme.hpp` — Current Theme API: unordered_map declarations, `c()`, `g()`, `dec()` inline functions
- `src/btop_theme.cpp` — `Default_theme`, `TTY_theme` maps, `generateColors()`, `generateGradients()`, `generateTTYColors()`, `loadFile()`, `setTheme()`
- `src/btop_config.hpp` — Reference pattern: `BoolKey`/`IntKey`/`StringKey` enums with `COUNT` sentinel, `constexpr` name tables

```cpp
inline constexpr ColorKey gradient_start_key(GradientKey gk) {
    constexpr std::array<ColorKey, static_cast<size_t>(GradientKey::COUNT)> starts = {
        ColorKey::temp_start, ColorKey::cpu_start, ColorKey::free_start,

        ColorKey::download_start, ColorKey::upload_start, ColorKey::process_start,
        ColorKey::proc_start, ColorKey::proc_color_start

    return starts[static_cast<size_t>(gk)];
}
```

Convert the implementation in `btop_theme.cpp` to use the new enum-indexed arrays:

1. **Replace map definitions** at the top of the `Theme` namespace:
   ```cpp
   std::array<string, static_cast<size_t>(ColorKey::COUNT)> colors{};
   std::array<std::array<int, 3>, static_cast<size_t>(ColorKey::COUNT)> rgbs{};
   std::array<std::array<string, 101>, static_cast<size_t>(GradientKey::COUNT)> gradients{};
   ```

2. **Convert `Default_theme`** from `unordered_map<string, string>` to a `constexpr std::array<std::pair<ColorKey, std::string_view>, N>` or keep it as-is for parsing simplicity and use `ColorKey` indexing in the iteration. The simplest approach: create a `constexpr` array of `std::pair<ColorKey, const char*>`:
   ```cpp
   static constexpr std::array default_theme_values = std::to_array<std::pair<ColorKey, const char*>>({
       {ColorKey::main_bg, "#00"},
       {ColorKey::main_fg, "#cc"},
       // ... all 38 entries
   });
   ```
   This replaces the `Default_theme` unordered_map. Similarly convert `TTY_theme`.

3. **Convert `generateColors()`** — the source parameter changes from `unordered_map<string, string>` to `std::unordered_map<string, string>` (still needed for theme file parsing which returns string keys). Iterate over `default_theme_values` by `ColorKey` instead of by string name. Use the `ColorKey` to index into `colors[]` and `rgbs[]` directly. For fallback logic (`meter_bg`, `process_start`, `graph_text`), use `ColorKey` enum comparisons. The `source` parameter can remain as `unordered_map<string, string>` since `loadFile()` returns string-keyed data from parsing .theme files. Look up the color name string using `color_key_names[static_cast<size_t>(key)]` to check if it exists in source.

4. **Convert `generateGradients()`** — iterate over `GradientKey` values (0..COUNT-1). For each gradient key, compute the _start, _mid, _end `ColorKey` offsets. First, insert the synthetic proc/proc_color entries into `rgbs[]` using their `ColorKey` indices:
   ```cpp
   rgbs[static_cast<size_t>(ColorKey::proc_start)] = rgbs[static_cast<size_t>(ColorKey::main_fg)];
   rgbs[static_cast<size_t>(ColorKey::proc_mid)] = {-1, -1, -1};
   // ... etc.
   ```
   Then loop over each `GradientKey`, look up the three ColorKey entries (_start, _mid, _end — they are contiguous: `gradient_start_key(gk) + 0/1/2`), compute the gradient array, and store in `gradients[static_cast<size_t>(gk)]`.

5. **Convert `generateTTYColors()`** — similar pattern: populate `colors[]` from `TTY_theme` array, iterate gradient keys to fill `gradients[]`.

6. **Convert `loadFile()`** — this function reads .theme files which use string keys. Keep the return type as `unordered_map<string, string>` since theme files are string-based. The validation check `Default_theme.contains(name)` becomes a check against `color_key_names` or `colorKeyFromString()`.

7. **Implement the string-to-enum lookup helpers:**
   ```cpp
   std::optional<ColorKey> colorKeyFromString(std::string_view name) {
       for (size_t i = 0; i < color_key_names.size(); ++i)
           if (color_key_names[i] == name) return static_cast<ColorKey>(i);
       return std::nullopt;
   }
   ```
   Same for `gradientKeyFromString`.

8. **Implement temporary string overloads** of `c()`, `g()`, `dec()`:
   ```cpp
   const string& c(const string& name) {
       if (auto key = colorKeyFromString(name)) return colors[static_cast<size_t>(*key)];
       static const string empty;
       return empty;
   }
   const std::array<string, 101>& g(const string& name) {
       if (auto key = gradientKeyFromString(name)) return gradients[static_cast<size_t>(*key)];
       // This should never happen at runtime — theme setup guarantees all gradients exist
       static const std::array<string, 101> empty{};
       return empty;
   }
   ```

9. **Update `setTheme()`** — the two lines at the bottom that access `colors.at("main_fg")` and `colors.at("main_bg")` become `colors[static_cast<size_t>(ColorKey::main_fg)]` and `colors[static_cast<size_t>(ColorKey::main_bg)]`.

10. **Keep `loadFile()` returning unordered_map<string,string>`** — this is the parsing interface for .theme files (string-keyed input). The `generateColors()` function accepts this and maps string keys to ColorKey via `colorKeyFromString()`.

Ensure `Default_theme` and `TTY_theme` references in `loadFile()` validation use the new constexpr array pattern. The `loadFile` function checks `Default_theme.contains(name)` to skip unknown keys — replace with `colorKeyFromString(name).has_value()`.
