**2. Add SoA sort helper function:**
```cpp
namespace {
    void soa_sort(vector<proc_info>& proc_vec, auto projection, bool ascending, int display_count) {
        thread_local vector<SortEntry> keys;
        keys.resize(proc_vec.size());

        // Extract keys
        for (uint32_t i = 0; i < proc_vec.size(); ++i)
            keys[i] = {static_cast<double>(projection(proc_vec[i])), i};

        // Determine sort range
        const int n = (display_count > 0 && display_count < static_cast<int>(keys.size()))
            ? display_count : static_cast<int>(keys.size());
        auto middle = keys.begin() + n;

        // Sort the compact key array
        if (ascending)
            std::partial_sort(keys.begin(), middle, keys.end(),
                [](const SortEntry& a, const SortEntry& b) { return a.key < b.key; });
        else
            std::partial_sort(keys.begin(), middle, keys.end(),
                [](const SortEntry& a, const SortEntry& b) { return a.key > b.key; });

        // Permute: build sorted front from index array
        thread_local vector<proc_info> sorted_front;
        sorted_front.clear();
        sorted_front.reserve(n);
        for (auto it = keys.begin(); it != middle; ++it)
            sorted_front.push_back(std::move(proc_vec[it->index]));

        // Move sorted entries to front of proc_vec
        for (size_t i = 0; i < sorted_front.size(); ++i)
            proc_vec[i] = std::move(sorted_front[i]);
    }
}
```

**3. Update proc_sorter switch cases** for numeric keys (cases 0, 3, 5, 6, 7):

For each numeric sort key, when `!tree && proc_vec.size() >= SOA_THRESHOLD`:
```cpp
case 6:  // cpu_p
    if (!tree && proc_vec.size() >= SOA_THRESHOLD)
        soa_sort(proc_vec, [](const proc_info& p) { return p.cpu_p; }, !reverse, display_count);
    else if (use_partial)
        rng::partial_sort(proc_vec.begin(), middle, proc_vec.end(), rng::less{}, &proc_info::cpu_p);
    else
        rng::stable_sort(proc_vec, rng::less{}, &proc_info::cpu_p);
    break;
```

Wait — the `reverse` flag semantics: when `reverse=true`, the sort order is "less" (ascending) for numeric keys like cpu_p (cases 6,7 — the "reverse" block sorts ascending so that highest CPU is at the bottom). When `reverse=false`, it sorts "greater" (descending, highest first). So pass `ascending = reverse` for cpu_p/cpu_c/mem/threads, and `ascending = !reverse` for pid. Check each case carefully against the existing stable_sort comparators.

**String sort keys (cases 1, 2, 4: name, cmd, user)** — leave unchanged. SoA doesn't help for string comparisons since comparison still chases pointers.

**Tree mode** — leave unchanged (`tree` already forces full stable_sort via the existing `use_partial` guard, and the SoA path checks `!tree`).

**"cpu lazy" rotation** — remains at the end, operates on the already-sorted front of proc_vec. Works correctly with SoA permutation since the front N entries are in sorted order.

**Key constraint:** The `soa_sort` function uses `std::partial_sort` internally (not `rng::partial_sort`), composing ALG-01 and ALG-03. When `display_count > 0`, only the top display_count keys are sorted; when `display_count <= 0`, all keys are sorted (full partial_sort where middle == end is equivalent to full sort).

**Note on thread_local:** The `thread_local` vectors avoid re-allocation each cycle. Only the runner thread calls proc_sorter, so there's no contention.
