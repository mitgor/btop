<objective>
Batch Linux /proc file reads via io_uring with automatic fallback to sequential POSIX I/O.

Purpose: The Linux process collection hot path reads /proc/[pid]/stat for every process every cycle — each requiring 3 syscalls (open + read + close). With 500 processes, that is 1500 syscalls per cycle just for stat files. io_uring batches all open+read+close chains into a single io_uring_submit() call, reducing syscall count from ~3N to ~1.

Output: ProcReader abstraction with io_uring backend + POSIX fallback, CMake integration, integration into process collection hot path, unit tests for fallback path.

IMPORTANT: This plan creates Linux-only source files. The development machine is macOS. All io_uring code MUST be behind `#ifdef USE_IO_URING` guards. The POSIX fallback path and tests MUST compile and run on macOS. The io_uring-specific code will only compile on Linux with liburing installed.

```cpp
namespace Tools {
    inline ssize_t read_proc_file(const char* path, char* buf, size_t buf_size) {
        int fd = ::open(path, O_RDONLY | O_CLOEXEC);
        if (fd < 0) return -1;
        ssize_t total = 0;
        while (static_cast<size_t>(total) < buf_size - 1) {
            ssize_t n = ::read(fd, buf + total, buf_size - 1 - total);
            if (n <= 0) break;
            total += n;
        }
        ::close(fd);
        buf[total] = '\0';
        return total;
    }
}
```

**src/linux/io_uring_reader.hpp:**
Create the ProcReader header. The entire file is Linux-only (only compiled when target is Linux), but the io_uring-specific members are behind `#ifdef USE_IO_URING`.

```cpp
#pragma once
#include <cstddef>
#include <cstdint>

#ifdef USE_IO_URING
#include <liburing.h>
#endif

namespace Linux {

class ProcReader {
public:
    struct ReadRequest {
        const char* path;     // e.g., "/proc/1234/stat"
        char* buffer;         // caller-owned buffer
        size_t buf_size;      // buffer capacity
        ssize_t result;       // bytes read on success, -1 on error
    };

    ProcReader();
    ~ProcReader();

    // Batch-read multiple /proc files. Returns number of successful reads.
    // Uses io_uring when available, otherwise sequential POSIX fallback.
    int submit_batch(ReadRequest* requests, int count);

    // Whether io_uring is active (for logging/benchmarking)
    bool using_io_uring() const;

    // Non-copyable
    ProcReader(const ProcReader&) = delete;
    ProcReader& operator=(const ProcReader&) = delete;

private:
    int sequential_fallback(ReadRequest* requests, int count);

#ifdef USE_IO_URING
    struct io_uring ring;
    bool uring_available = false;
    static constexpr unsigned RING_SIZE = 4096; // power-of-2, supports up to ~1365 files (3 SQEs each)
    static constexpr int MAX_BATCH = 1024;      // max files per submit (3072 SQEs)

    bool init_uring();
    int uring_batch(ReadRequest* requests, int count);
#endif
};

} // namespace Linux
```

**src/linux/io_uring_reader.cpp:**
Implementation with two paths:

1. **POSIX fallback (`sequential_fallback`):** Calls `Tools::read_proc_file()` for each request sequentially. This always works.

2. **io_uring path (`uring_batch`)** — only compiled with `#ifdef USE_IO_URING`:
   - Constructor calls `init_uring()` which does:
     - `io_uring_queue_init(RING_SIZE, &ring, 0)` — if returns < 0, set `uring_available = false`
     - Probe for OPENAT, READ, CLOSE opcodes via `io_uring_get_probe_ring()`
     - If all supported, set `uring_available = true`
   - `submit_batch()`: if `uring_available` and count > 0, call `uring_batch()`, else `sequential_fallback()`.
   - `uring_batch()`:
     - If count > MAX_BATCH, process in batches of MAX_BATCH to avoid ring exhaustion.
     - For each file, prepare 3 linked SQEs:
       a. **OPENAT:** `io_uring_prep_openat(sqe, AT_FDCWD, path, O_RDONLY | O_CLOEXEC, 0)`. Set `sqe->flags |= IOSQE_IO_LINK`.
       b. **READ:** `io_uring_prep_read(sqe, 0, buffer, buf_size - 1, 0)`. Use `IOSQE_IO_HARDLINK` (not IO_LINK) for the read->close link to prevent short-read chain cancellation (Pitfall 1 from research). The fd from the linked OPENAT is automatically used.
       c. **CLOSE:** `io_uring_prep_close(sqe, 0)`. No link flag on the last SQE in the chain.
     - Use `io_uring_sqe_set_data(sqe, user_data)` to tag each SQE with its request index and operation type (use a small struct or encode index+op in a uint64_t).
     - Call `io_uring_submit(&ring)` once for all chains.
     - Harvest completions: loop `count * 3` times calling `io_uring_wait_cqe(&ring, &cqe)`. For READ CQEs, store `cqe->res` as the bytes-read result. For OPENAT CQEs, the `cqe->res` is the fd (needed for close if not using linked). For CLOSE CQEs, just consume. Call `io_uring_cqe_seen(&ring, cqe)` after each.
     - Null-terminate each buffer: `request.buffer[request.result] = '\0'`.
     - If any step fails (io_uring_get_sqe returns NULL), fall back to `sequential_fallback()` for remaining requests.
   - Destructor: if `uring_available`, call `io_uring_queue_exit(&ring)`.

**IMPORTANT LINKED SQE DETAIL:** For linked open+read+close, the read SQE's fd field is set to the result of the preceding open. With IOSQE_IO_LINK, io_uring handles this automatically when the open completes — the kernel fills in the fd for the next linked operation. Set the read SQE's fd field to `0` as a placeholder; the link mechanism replaces it. Same for close. However, double-check liburing docs — if this automatic fd passing requires `IOSQE_FIXED_FILE`, add it. If the linked fd passing does NOT work as expected, fall back to a two-phase approach: submit all OPENATs, harvest fds, then submit READ+CLOSE pairs.

**Overview:** The process collection loop in `src/linux/btop_collect.cpp` iterates over all PIDs and reads `/proc/[pid]/stat` for each one (line ~3268). Currently this is sequential — one `read_proc_file()` per PID. Refactor to use ProcReader for batch submission of all stat file reads.

**Strategy:** Two-pass approach in the process loop:
1. **Pass 1 (collect paths):** Iterate PIDs, build path strings and allocate buffers for each PID's `/proc/[pid]/stat` read. Handle cache-miss reads (comm, cmdline, status) sequentially as before (per Pitfall 6 — only batch the hot path).
2. **Pass 2 (batch read):** Submit all stat ReadRequests via ProcReader::submit_batch().
3. **Pass 3 (parse results):** Iterate results and parse stat fields as before.

**Detailed changes:**

1. Add `#include "io_uring_reader.hpp"` at top of file (this is already a Linux-only file).

2. In the process collection function (the one containing the PID iteration loop around line 3150+), create a static ProcReader instance:


3. Before the PID loop, allocate batch arrays. Use the existing `pid_list` size to pre-allocate:

   - Call `proc_reader.submit_batch(reqs, count)`.
