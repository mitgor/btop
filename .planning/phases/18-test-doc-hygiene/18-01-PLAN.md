  **Goal:** Make push_back() a no-op on zero-capacity RingBuffer, matching the test expectation and following standard container semantics.

  **Step 1 — Fix push_back() in src/btop_shared.hpp (line 255-267):**

  Change:
  ```cpp
  void push_back(const T& value) {
      if (capacity_ == 0) {
          resize(1);
      }
      if (size_ < capacity_) {
  ```
  To:
  ```cpp
  void push_back(const T& value) {
      if (capacity_ == 0) return;
      if (size_ < capacity_) {
  ```

  This makes push_back a silent no-op when capacity is zero, which is the correct behavior. A zero-capacity buffer has no storage allocated — pushing into it should do nothing, just like pop_front on an empty buffer is a no-op (line 270).

  **Why this is safe:** Verified by grep that all production usage of RingBuffer falls into one of:
  1. Constructed with explicit capacity > 0 (e.g., `RingBuffer<long long> buf(3);`)
  2. Default-constructed, then `resize(N)` called before any push_back
  3. Default-constructed data members (`.temp{}`, `.io_read{}`, etc.) that are resized during Draw initialization before data collection pushes into them

  No production code path calls push_back on a zero-capacity buffer expecting it to auto-resize. The auto-resize was defensive code added in Phase 4 that created the test mismatch.
