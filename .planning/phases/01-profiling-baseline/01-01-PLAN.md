2. **Create benchmarks/CMakeLists.txt:** Follow the existing tests/CMakeLists.txt pattern. Use FetchContent to fetch nanobench v4.3.11 (single-header, GIT_SHALLOW TRUE). Create a helper library `libbtop_bench` that links `libbtop` and `nanobench`, with include directories pointing to `${PROJECT_SOURCE_DIR}/src`. Add `btop_bench_strings` executable from `bench_string_utils.cpp`.

```cmake
include(FetchContent)
FetchContent_Declare(
    nanobench
    GIT_REPOSITORY https://github.com/martinus/nanobench.git
    GIT_TAG v4.3.11
    GIT_SHALLOW TRUE
)
FetchContent_MakeAvailable(nanobench)

add_library(libbtop_bench INTERFACE)
target_include_directories(libbtop_bench INTERFACE ${PROJECT_SOURCE_DIR}/src)
target_link_libraries(libbtop_bench INTERFACE libbtop nanobench)

add_executable(btop_bench_strings bench_string_utils.cpp)
target_link_libraries(btop_bench_strings PRIVATE libbtop_bench)
```

3. **Create benchmarks/bench_string_utils.cpp:** Implement micro-benchmarks for all string utility hot functions. Structure:
   - `#define ANKERL_NANOBENCH_IMPLEMENT` then `#include <nanobench.h>`
   - Include `btop_tools.hpp`
   - Parse `--json` CLI flag (argv[1]) to switch between console and JSON output
   - Prepare realistic test data: a colored string with ANSI escape codes (80+ chars with `\033[38;5;Nm` sequences) and a plain UTF-8 string (~200 chars)
   - Benchmark these functions with `bench.run(name, lambda)` using `doNotOptimizeAway`:
     - `Fx::uncolor` on the colored string
     - `Tools::ulen` on the plain string
     - `Tools::uresize` to 50 chars
     - `Tools::ljust` padding to 100
     - `Tools::rjust` padding to 100
     - `Tools::cjust` padding to 100
     - `Mv::to(10, 50)` cursor movement
   - Set `bench.warmup(100)` and `bench.relative(true)`
   - For JSON output: `bench.output(nullptr)` then `bench.render(ankerl::nanobench::templates::json(), std::cout)`
   - Return 0

Note: The benchmark must NOT call any btop initialization functions (Shared::init, Config::load, etc.) since string utilities are standalone. This keeps the benchmark fast and dependency-free.

1. **Create benchmarks/bench_draw.cpp:** Micro-benchmarks for draw functions. This is more complex because draw functions depend on initialized state (Config, Theme, terminal dimensions).

   Structure:
   - Include nanobench (do NOT `#define ANKERL_NANOBENCH_IMPLEMENT` -- already done in bench_string_utils.cpp; instead include the header only). Actually, since each benchmark is a separate executable, DO define `ANKERL_NANOBENCH_IMPLEMENT` in each.
   - Include `btop_draw.hpp`, `btop_shared.hpp`, `btop_config.hpp`, `btop_theme.hpp`, `btop_tools.hpp`
   - Parse `--json` flag
   - **Minimal initialization:** Before benchmarking, initialize the minimum state needed:
     - Set `Term::width = 200` and `Term::height = 50` directly (these are in btop_shared.hpp as global vars)
     - Call `Config::load(...)` with a default config path (or empty path to use defaults)
     - Call `Theme::setTheme()` to load default theme
     - Initialize Shared state enough for draw functions to work
   - If initialization proves too complex (draw functions may crash without full init), fall back to benchmarking individual components:
     - Graph::operator() with synthetic data
     - Meter::operator() with synthetic percentage values
     - The string composition in draw functions (fmt::format calls)
   - Benchmark with `doNotOptimizeAway`
   - JSON output support identical to bench_string_utils.cpp

   **IMPORTANT:** If full draw function benchmarking requires too much init state, document what was achieved and what needs the `--benchmark` mode (Plan 02) for realistic measurement. Partial coverage is acceptable for micro-benchmarks -- the macro-benchmark covers the full path.

2. **Create benchmarks/bench_proc_collect.cpp:** Linux-only micro-benchmark for Proc::collect.

   Structure:
   - Guard entire file with `#ifdef __linux__` (and provide a stub main for other platforms)
   - Include nanobench, `btop_shared.hpp`
   - Parse `--json` flag
   - Initialize minimal state for Proc::collect (this reads real /proc data, so needs Shared::init or equivalent)
   - Benchmark `Proc::collect(false)` (no need for sorting in benchmark)
   - If init is too complex, benchmark the parsing subroutines in isolation:
     - Read `/proc/self/stat` into a string, benchmark parsing that string
     - Read `/proc/stat` into a string, benchmark parsing CPU usage data
   - JSON output support
